# 上下文超限压缩重试方案存档

日期：2026-04-18

## 问题背景

网关未必能提前准确知道上游模型的真实 token 上限。

因此，一个更现实的策略不是“事先精确预算”，而是：

1. 先按原始请求正常转发
2. 只有当上游明确返回“上下文过长 / token 超限”类错误时
3. 再由网关自动压缩请求内容
4. 压缩后重新请求同一个模型
5. 如果仍然超限，则继续压缩并重试
6. 直到成功，或达到压缩/重试上限后返回失败

## 核心思路

这是一个基于错误触发的自动压缩重试机制：

- 正常请求先发
- 命中超限错误才介入
- 由网关自动压缩历史上下文
- 压缩后重新发起请求
- 必要时进行多轮压缩重试

## 只应命中的错误类型

这个机制不能对所有错误都生效，只能针对“上下文超限类”错误。

典型可识别错误：

- `context_length_exceeded`
- `maximum context length`
- `prompt is too long`
- `too many input tokens`
- `input too large`
- `token limit exceeded`

不应进入压缩重试的错误：

- `401`
- `403`
- `429`
- `503`
- 模型不存在
- 上游 provider 内部错误

## 压缩对象的原则

不能无脑压缩整段请求。

应优先压缩：

- 较早的历史对话
- 较长的旧 user / assistant 内容

应尽量保留：

- `system`
- 当前用户最新一轮问题
- 最近若干轮对话
- 最近的 `tool_calls`
- 最近的 `tool` 结果

## 重试边界

必须设置上限，避免死循环。

例如：

- 最多自动压缩重试 3 次
- 或压缩到只剩 `system + 最近几轮`
- 超过上限后直接返回错误

## 关于流式请求

这个机制更适合 `stream=false`。

原因：

- 流式响应一旦开始向客户端发送字节，就很难优雅回滚后再自动重试
- 所以“超限后压缩再重试”更适合在响应开始前完成

## 关于客户端通知

希望做到：

- 客户端不用额外操作
- 也能知道网关正在执行“压缩并重试”

这个需求分情况：

### 1. 非流式请求

基本无法实时通知。

原因：

- 同一个 HTTP 响应在非流式模式下只能最终一次性返回
- 中途发生了什么，客户端通常看不到

### 2. 流式请求

理论上可以通知，但要看客户端是否支持显示。

可选方式：

#### 方式 A：SSE 注释或自定义事件

示例：

```text
: gateway compressing and retrying
```

或：

```text
event: gateway_status
data: {"stage":"compress_retry","attempt":1}
```

优点：

- 不污染模型正文

缺点：

- 很多 OpenAI-compatible 客户端会直接忽略
- `opencode` 这类客户端大概率不会直接展示

#### 方式 B：把提示塞进 assistant content

示例：

```text
[gateway] 上下文超限，正在压缩重试...
```

优点：

- 客户端几乎一定能看到

缺点：

- 会污染模型回答正文
- 还可能污染后续上下文历史

因此不推荐

#### 方式 C：最终响应头或元信息

示例：

- `x-gateway-context-compressed: 1`
- `x-gateway-context-retries: 2`
- `x-gateway-request-id: ...`

优点：

- 干净
- 不污染正文

缺点：

- 客户端处理中通常看不到
- 大多数通用客户端也不会主动展示

## 对 opencode 这类客户端的现实判断

如果要求同时满足：

- 客户端不用改
- 用户中途能看到“正在压缩重试”
- 不污染模型正文

那么在 OpenAI-compatible 协议下，基本很难优雅实现。

最现实的结论是：

- 非流式：无法实时展示
- 流式：只有自定义 SSE 事件才比较干净，但客户端大概率不会显示
- 最稳但最脏的方式是把提示直接写进正文

## 当前结论

“超限后由网关自动压缩，再重新请求，仍超则继续压缩重试”这个方向是成立的。

但客户端通知能力要分开看：

- 压缩重试：网关可以做
- 客户端无改动实时感知：很难保证

更合理的落地方向是：

- 先实现网关自动压缩重试
- 再根据目标客户端能力决定是否增加通知机制
